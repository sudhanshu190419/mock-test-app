4. LiveKit Architecture
Token Generation Flow
Client → supabase.functions.invoke('livekit-token') → Edge Function
  ↓ Validate caller JWT (Supabase auth)
  ↓ Extract: roomName, participantName, role
  ↓ Generate LiveKit AccessToken with LiveKit Server SDK
  ↓ Return { token: "eyJ...", url: "wss://project.livekit.cloud" }
Client → LiveKit Room.connect(url, token)
Edge Function Details
- Location:  supabase/functions/livekit-token/ 
- Environment: Supabase Edge Function (Deno runtime)
- Required Env Vars:  LIVEKIT_API_KEY ,  LIVEKIT_API_SECRET ,  LIVEKIT_URL 
- Accepts:  { roomName, participantName, role }  from client
- Returns:  { token, url }  with LiveKit JWT
- Security: Validates caller's Supabase auth JWT + injects caller metadata

1. Room lifecyle is per-hook, not provider-based. Each screen creates its own  Room  instance.
2. No LiveKitProvider context — Room state is managed locally in  useLiveKit  hook.
3. Audio configuration is comprehensive for Android (VoIP quality, loudspeaker routing).
4. Diagnostics module is extensive — logs audio state, Room state, WebRTC internals.
5. Reconnection logic: Built into  livekit-client  (Room handles auto-reconnect). UI tracks  reconnecting  state but has no explicit reconnection UI.

6. Authentication & Authorization
Authentication Flow
Teacher Website:
  Phone/Password → Supabase Auth → session/user → loadTeacherProfileDetails()
  Fallback: Demo mode with localStorage mock
 
Student App:
  Phone OTP → Supabase Auth → auth state listener → profile load
Role Detection
-  profiles.role  enum:  admin  |  teacher  |  student  (+  user  added in migration 045)
- Role checked on profile load, drives UI rendering
- Teacher website checks  role === 'teacher'  vs  role === 'admin' 
Institute (Tenant) Isolation
-  get_my_institute_id()  function returns institute_id for current user
- Every domain table has  institute_id  column
- RLS policies filter by  institute_id = get_my_institute_id() 
RLS Policies for Live Tables
- Currently NO explicit RLS policies for live tables in migration 021
- Live tables have  institute_id  columns but lack granular RLS
- RISK: Without explicit policies, default Supabase RLS blocks all access or allows all
Required for Live Classes
- Teacher should only see/modify their own  live_classes  (filter by  teacher_id )
- Students should only see  live_classes  linked to their batches
- Student access to  live_sessions  should only return  participant_token , never  host_token

Summary
The architecture is solid at the database level — Domain 04's schema is production-ready with proper relationships, indexes, and constraints. The backend Edge Function for LiveKit tokens is implemented. The student app has a working LiveKit POC that can be refactored into production code.
The single biggest gap is the teacher website, which has no LiveKit client integration at all. It simulates a live class using raw browser WebRTC without connecting to any video provider. The  live_classes  lifecycle methods exist but don't create LiveKit rooms or generate tokens.
The second gap is the missing abstraction layer — both the teacher website (when it's built) and the student app would benefit from a  VideoProvider  interface that allows swapping LiveKit for another provider without UI changes.
No new database tables are needed for Phase 1. The existing Domain 04 schema supports the full live class lifecycle. The implementation effort is primarily in the service/UI layer.
